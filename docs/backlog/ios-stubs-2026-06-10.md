## IOS-SYNC-001 · ParkingSyncScheduler → Firestore ✅

**Ticket:** `IOS-SYNC-001`
**Prioridad:** 🔴 P0 | **Esfuerzo:** Grande
**Estado:** ✅ Done — verificado 2026-06-10 (ya estaba implementado)

### Resolución

Auditoría inicial confundió `StubParkingSyncScheduler.kt` (código muerto, nunca registrado en Koin) con el scheduler activo. El **real** `IosParkingSyncScheduler` ya existía:

- Coroutine scope (`SupervisorJob + Dispatchers.Default`)
- Retry exponencial (5 intentos, backoff 30s × 2ⁿ)
- 3 métodos cableados a `RemoteUserProfileDataSource` (GitLive Firebase, multiplatform — soporta iOS)
- Registrado en `IosDetectionModule.kt:31`

### Cleanup

Borrado el dead-code `composeApp/src/iosMain/.../ios/stub/StubParkingSyncScheduler.kt`.

### Limitación residual (no bloqueante para beta)

Sin persistencia cross-process-death — si la app muere mientras hay un write pendiente, se pierde. Integración con `BGProcessingTaskRequest` queda como mejora futura tracked en el docstring de `IosParkingSyncScheduler.kt`. **No es bloqueante** porque en el flujo típico (iOS mantiene el proceso vivo unos segundos tras background) los writes completan antes del kill.

## IOS-NOTIF-001 · Notificaciones faltantes + UNUserNotificationCenterDelegate ✅

**Ticket:** `IOS-NOTIF-001`
**Prioridad:** 🟠 P1 | **Esfuerzo:** Medio
**Estado:** ✅ Done 2026-06-10
**Archivos:**
- `composeApp/src/iosMain/.../notification/IosAppNotificationManagerImpl.kt` — añadidos 3 overrides + nueva category
- `composeApp/src/iosMain/.../notification/IosNotificationActionHandler.kt` — nuevo
- `composeApp/src/iosMain/.../MainViewController.kt` — instala el delegate

### Fix aplicado

**A. Métodos añadidos** con copia inglesa consistente con Android `notif_*`:

| Método | Notification ID | Category |
|--------|----------------|----------|
| `showParkingSavedConfirm(parkingId, vehicleName, lat, lon)` | `PARKING_CONFIRMATION_NOTIFICATION_ID` (reemplaza prompt — REFACTOR-300) | `CATEGORY_PARKING_SAVED_CONFIRM` |
| `showHomeParkingLeft(label, lat, lon)` | `HOME_PARKING_NOTIFICATION_ID` | (sin acciones) |
| `showConfirmationFailed()` | `CONFIRMATION_FAILED_NOTIFICATION_ID` | (sin acciones) |

`parkingId` viaja en `content.userInfo[EXTRA_PARKING_ID]`.

**B. Action routing.** Nuevo `IosNotificationActionHandler : NSObject, UNUserNotificationCenterDelegateProtocol`:

| Action ID | Handler |
|-----------|---------|
| `ACTION_CONFIRMED` | `coordinator.onUserConfirmedParking()` |
| `ACTION_DENIED` | `coordinator.onUserDeniedParking()` |
| `ACTION_ACK` | `notificationPort.dismiss(PARKING_CONFIRMATION_NOTIFICATION_ID)` |
| `ACTION_REVERT` | `revertParkingUseCase(parkingId)` en `SupervisorJob` scope |

El handler se mantiene vivo como `private var` en `MainViewController.kt` (delegate de iOS es weak; sin esto el routing se desactivaría silenciosamente al GC).
