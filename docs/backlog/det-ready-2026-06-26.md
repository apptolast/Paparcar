## 0. Decisiones de producto (cerradas antes de tocar código)

### Decisión A — Gating de permisos: **foreground-gate, NO all-gate**
- **Hard-require solo el tier CORE**: ubicación foreground + notificaciones. Sin ubicación
  el mapa de plazas no tiene sentido → ese muro es honesto y el usuario lo entiende.
- **Tier PRODUCER** (background location + activity recognition): se pide **contextual**,
  en el momento de valor (registro de coche / primer aparcamiento / CTA "Activar detección"),
  y es **re-pedible** desde el banner de Home. Nunca bloquea Home.
- **Tier OPTIONAL** (BT connect, battery exemption, OEM autostart): oportunista.
- Razón: Android 11+ ni siquiera permite pedir background+foreground juntos; "obligar todo"
  es técnicamente imposible y maximiza el drop-off install→active + riesgo de rechazo en Play.

### Decisión B — **NO** gatear "ver plazas libres" tras tener detección activa
- El mapa de plazas es el lado **Consumidor** y solo necesita foreground location.
- Gatearlo tras background-detection castiga al usuario antes de darle valor y rompe el
  embudo. El free-rider problem se resuelve con **nudge + reciprocidad**, no con muro.
- En su lugar: banner persistente honesto cuando la detección está OFF
  (`Blocked`/`Disabled`) que invita a activarla.

### Decisión C — Reframe de la auto-detección: **"automatiza tu aparcamiento"**, no "publica plazas"
La detección se vende como automatización personal que da valor **aunque nunca publiques**:
1. **Recuerda dónde aparcaste** (find-my-car) — valor standalone inmediato.
2. **Historial automático** de aparcamientos (sin registrar nada a mano).
3. **Duración / zona** del aparcamiento.
4. Y *además*, publica tu plaza para la comunidad (reciprocidad).
- Copy de onboarding + empty states deben liderar con (1)+(2), no con "ayuda a la comunidad".

### Decisión D — Estado idle "sin geofence ni coche aparcado" se descompone en 3 sub-estados
1. **Armado y a la espera** (permisos OK, coche de parking activo, no aparcado)
   → tranquilizador, sin acción: *"Listo. Detectaré cuando aparques."*
2. **Bloqueado** (falta permiso PRODUCER) → accionable: *"Detección en pausa — actívala."*
3. **No aplicable** (sin coche / coche moto-bici) → CTA onboarding: *"Añade un coche."*

## Modelo central nuevo: `DetectionReadiness` (commonMain/domain)

```
sealed class DetectionReadiness {
    data class Disabled(reason: DisabledReason)        // NO_VEHICLE | NON_PARKING_VEHICLE
    data class Blocked(missing: Set<RequiredPermission>) // falta CORE o PRODUCER
    data class Ready(strategy: ParkingStrategy)          // armado, idle  ← "sin geofence"
    data class Monitoring(strategy: ParkingStrategy)     // siguiendo trayecto AHORA
    data class Parked(session: UserParking)              // aparcado, geofence vigilando salida
}
```
- `RequiredPermission` enum domain-pure: `FOREGROUND_LOCATION`, `BACKGROUND_LOCATION`,
  `ACTIVITY_RECOGNITION`, `NOTIFICATIONS`. Cada una con su `tier` (CORE | PRODUCER).
- `Parked` es el único estado con geofence; "sin geofence" = `Ready`. Eso responde
  literalmente la pregunta original de producto.
- `VehicleMonitoringStatus` (per-vehículo, chips de coche) **se conserva** — distinto scope.
  `DetectionReadiness` es global (coche activo). Se documenta la relación, no se fusionan.

**Modelado (cerrado):** `DetectionReadiness` es un **campo `Immutable` en `HomeState`**, NO una
variante de `HomeMode` ni un estado top-level aparte. Razón: `HomeMode`
(Browse/Reporting/AddingZone/AddingParking) es **qué hace el usuario** (interacción, mutuamente
excluyente); `DetectionReadiness` es **el estado del sistema de detección de fondo** — eje
**ortogonal**. Coexisten (Browse + Ready, Browse + Blocked, AddingParking + Ready…). Meterlo en
`HomeMode` provocaría explosión combinatoria. El campo **reemplaza** el actual
`HomeState.allPermissionsGranted: Boolean` (línea 46), que es su precursor pobre. El evento
transitorio `pendingParkingGps` (caso MEDIUM "pregunta al usuario") se mantiene aparte: es un
evento, no el estado estable de readiness.
